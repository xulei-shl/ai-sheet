# 自我提升

> **何时使用：**
>
> - Python 脚本执行后，过程中出现过错误、重试、修复或用户纠正
> - 同一类问题重复出现，或同一脚本自动修复 2 次及以上
> - 发现了可复用的 Excel / Python 数据处理经验
> - 发现当前 `.pi/AGENTS.md` 或某个技能文档与实际工作流不一致
> - 用户明确指出 agent 的行为需要调整
> - Python 处理失败达到重试上限，需要沉淀失败原因或排查路径

---

## 核心原则

1. **先询问，后总结**：不得在未询问用户的情况下启动文档修改。
2. **先草稿，后写入**：不得直接修改 `.pi/AGENTS.md` 或技能文件，必须先给出建议和草稿。
3. **最终确认后才写入**：只有用户明确确认草稿后，才能更新对应 Markdown 文件。
4. **只沉淀可复用经验**：不要为了记录而记录；一次性、无复用价值的问题不写入。
5. **优先更新具体技能**：Python 执行相关经验优先写入 `python-processing`，通用行为规则才写入 `AGENTS.md`。
6. **保持简洁**：新增内容应短小、明确、可执行，避免让规则文件膨胀。

---

## 工作流程

### Step 1：判断是否值得沉淀

完成任务或 Python 执行流程后，检查是否出现以下情况：

- 是否发生了 Python 报错并进行了修复？
- 是否同一问题重试了 2 次及以上？
- 是否发现了新的常见错误模式？
- 是否用户纠正了 agent 的行为？
- 是否发现技能文档缺少关键步骤？
- 是否发现 `.pi/AGENTS.md` 的规则与实际需要不一致？
- 是否出现了以后可能重复遇到的数据坑，例如列名不匹配、路径转义、编码、日期格式、前导零、Excel 文件占用等？

如果没有以上情况，则不需要触发自我提升。

---

### Step 2：询问用户是否总结经验

如果值得沉淀，先向用户询问，例如：

> 这次处理过程中遇到了 `KeyError`，原因是脚本假设了列名，但实际 Excel 列名不同。  
> 我通过先打印 `df.columns.tolist()` 再修正列名解决了问题。  
> 是否需要我把这条经验总结到项目技能文档中，方便以后自动避免？

用户拒绝时，停止流程，不写入任何文档。

---

### Step 3：分析经验应写入哪里

按以下规则选择目标文件：

| 经验类型 | 推荐写入位置 |
|---|---|
| Agent 通用行为规则、交互规则、必须确认的操作 | `.pi/AGENTS.md` |
| Python 脚本编写、执行、调试、依赖、pandas/openpyxl 经验 | `.pi/skills/python-processing/SKILL.md` |
| 自我提升流程本身需要调整 | `.pi/skills/self-improvement/SKILL.md` |
| 某个未来新增专项流程 | 新建或更新对应 `.pi/skills/<skill-name>/SKILL.md` |

优先选择最具体的技能文件，避免把所有细节都塞进 `.pi/AGENTS.md`。

---

### Step 4：给出修改建议和草稿

在写入前，必须展示：

1. **建议写入位置**
2. **原因**
3. **拟新增或修改的 Markdown 草稿**
4. **是否会替换已有内容，还是追加新内容**

示例：

```markdown
建议写入：`.pi/skills/python-processing/SKILL.md` 的“常见错误模式”

原因：这是 Python 读取 Excel 时常见的列名不匹配问题，属于可复用调试经验。

拟新增：

| `KeyError` 且列名看似存在 | Excel 列名包含空格、换行或不可见字符 | 先打印 `repr` 形式的列名，例如 `[repr(c) for c in df.columns]`，再执行 `df.columns = df.columns.str.strip()` |
```

然后询问用户是否确认写入。

---

### Step 5：根据用户反馈迭代草稿

如果用户提出修改意见：

1. 不要立即写入；
2. 先根据用户意见更新草稿；
3. 再次请求确认；
4. 直到用户明确确认。

---

### Step 6：最终确认后写入文件

只有用户明确表示“确认”“可以写入”“按这个更新”等，才允许使用 `edit` / `write` 工具更新文件。

写入时遵循：

- 修改最小范围；
- 优先更新已有条目，避免重复；
- 保持 Markdown 表格格式正确；
- 不写入敏感数据、用户文件路径、真实业务数据；
- 不记录一次性上下文，只记录可复用规则。

---

### Step 7：写入后总结变更

更新完成后，向用户说明：

- 修改了哪个文件；
- 新增或调整了什么规则；
- 后续 agent 会在什么情况下受益；
- 如适用，提醒开发模式下 `.pi/` 资源复制机制可能需要同步到 `app_data_dir/.pi/`。

示例：

```
已更新 .pi/skills/python-processing/SKILL.md，在常见错误模式中新增了“列名包含不可见字符导致 KeyError”的处理方法。

以后 Python 处理 Excel 时，如果遇到列名匹配失败，会优先打印 repr 形式列名并清理空白字符。
```

---

## 输出格式

触发自我提升时，优先使用以下格式：

```
## 可沉淀经验

- 现象：
- 根因：
- 修复方式：
- 是否可复用：
- 建议写入：
- 草稿：

是否需要我将这条经验写入上述文档？
```

---

## 不应沉淀的情况

不要为以下内容更新文档：

- 仅与某个用户私有文件有关的一次性路径
- 不可复用的临时数据值
- 已有规则已经覆盖的问题
- 没有验证过的猜测
- 用户未确认的总结
- 会泄露隐私、API Key、文件路径或业务数据的内容

---

## 质量检查

写入前后检查：

1. 新内容是否具体、可执行？
2. 是否和现有规则重复或冲突？
3. 是否放在了最合适的文件？
4. 是否经过用户最终确认？
5. 是否避免了敏感信息？
6. 是否足够简洁？

---

## 与 Python 处理技能的关系

`python-processing` 技能负责完成 Python 脚本编写、执行、调试和 Excel 写入。

本技能只在以下阶段介入：

- Python 执行成功后，发现过程中有可复用经验；
- Python 执行失败或达到重试上限后，需要总结失败原因；
- 用户纠正了 Python 处理流程；
- 发现 `python-processing` 技能文档需要补充常见错误或最佳实践。

本技能不替代 Python 调试本身。
